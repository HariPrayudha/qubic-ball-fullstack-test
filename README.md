# Qubic Support Ticket System

A small support-ticket application where **users** open tickets and **admins** manage them
(update status, reply, filter). Built for the Qubic Ball Full-Stack technical assessment.

The repository contains three independent applications:

| App | Folder | Stack | Purpose |
|-----|--------|-------|---------|
| **Main API** | [`backend/`](backend/) | Laravel 13 · PHP 8.3 · MySQL · Sanctum | Tickets, auth, RBAC |
| **Frontend** | [`frontend/`](frontend/) | Next.js 16 · React 19 · TypeScript · Tailwind 4 · shadcn/ui | User & admin UI |
| **Stats service** | [`stats-service/`](stats-service/) | Node.js · TypeScript · Express 5 | Second backend: ticket statistics endpoint |

> **Second backend technology:** the main API is PHP/Laravel, so the required "second technology"
> task is implemented in **Node.js/TypeScript** as a standalone statistics endpoint
> (`GET /stats`). It reads the same database read-only and does not depend on the main app.

## Architecture

```
Browser
  │
  ▼
Next.js 16 (frontend, :3000)
  │   • React Server pages + Client Components
  │   • TanStack Query (data/cache/loading/error), React Hook Form + Zod (validation)
  │   • Route Handlers act as a thin proxy that attaches the Sanctum token
  │     from an httpOnly cookie → the token never reaches browser JavaScript
  ▼
Laravel 13 API (backend, :8000)
  │   Controller → FormRequest (validation) → Service (business logic)
  │            → Policy (RBAC) → Eloquent Model → API Resource (JSON shape)
  ▼
MySQL 8  ◄──── read-only ──── Node/TS stats service (:4000)
```

**Why an httpOnly-cookie proxy?** The Sanctum bearer token is stored in an httpOnly cookie set by
Next.js Route Handlers. The browser never sees the token (mitigates XSS token theft); client-side
TanStack Query calls the same-origin `/api/backend/*` proxy, which forwards to Laravel with the
`Authorization: Bearer` header read from the cookie.

**Backend layering** keeps responsibilities separated: thin controllers, `FormRequest` classes for
input validation, a `TicketService` for business logic (e.g. syncing `resolved_at` with status), a
`TicketPolicy` for authorization, and API Resources for a stable JSON contract.

### Frontend structure

```
frontend/src/
├── app/
│   ├── (auth)/            # route group: centered, navbar-less shell
│   │   ├── login/ · register/
│   ├── (app)/             # route group: navbar + page container
│   │   └── tickets/       # the single application screen
│   └── api/               # Route Handlers (server-side only)
│       ├── session/       # login / me / logout — sets the httpOnly cookie
│       └── backend/[...path]/  # authenticated proxy to Laravel
├── components/            # ui/ (shadcn primitives) + feature components
└── lib/                   # api client, query hooks, types, helpers
```

Creating and viewing a ticket are **modals** on the ticket list rather than separate pages, so the
list stays in context and no navigation state is lost. `lib/hooks.ts` centralises the TanStack
Query keys and cache invalidation; `lib/client.ts` centralises fetching and error shaping
(`ApiError` carries the field-level 422 messages that the forms display).

### Role-based features

The brief splits features by role, and the UI follows it:

| Feature | Customer | Admin |
|---|:--:|:--:|
| View own tickets / list | ✔ | ✔ (all tickets) |
| Create a ticket | ✔ | — |
| View ticket detail + responses | ✔ | — |
| Filter by status | — | ✔ |
| Update status · add a response | — | ✔ |

Authorization for the destructive/privileged actions is enforced server-side by `TicketPolicy`
(a customer calling them gets **403**); the role split above is the matching UI presentation.

## Setup & run

### 1. Database

Create the two databases (app + test) in MySQL:

```sql
CREATE DATABASE qubic_tickets      CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE DATABASE qubic_tickets_test CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

### 2. Backend — Laravel API (port 8000)

```bash
cd backend
cp .env.example .env
composer install
php artisan key:generate
php artisan migrate --seed     # run migrations + sample data
php artisan serve --port=8000
```

### 3. Stats service — Node/TS (port 4000)

```bash
cd stats-service
cp .env.example .env
npm install
npm run dev                    # http://localhost:4000/stats
```

### 4. Frontend — Next.js (port 3000)

```bash
cd frontend
cp .env.example .env.local
npm install
npm run dev                    # http://localhost:3000
```

Open **http://localhost:3000** and log in with a demo account below.

## Database migrations

```bash
cd backend
php artisan migrate             # apply migrations
php artisan migrate:fresh --seed  # drop, re-migrate, re-seed (resets demo data)
```

**Schema & relationships**

- `users` — `id, name, email, role(user|admin), …`
- `tickets` — `id, user_id → users, subject, description, status(open|in_progress|resolved), resolved_at, …`
- `ticket_responses` — `id, ticket_id → tickets, user_id → users, message, …`

`User hasMany Ticket` · `Ticket hasMany TicketResponse` · both responses/tickets cascade on delete.

## API reference (main backend)

All routes are prefixed with `/api`. Authenticated routes require `Authorization: Bearer <token>`.

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| POST | `/auth/register` | – | Register, returns user + token (201) |
| POST | `/auth/login` | – | Log in, returns user + token (200) |
| POST | `/auth/logout` | ✔ | Revoke current token (204) |
| GET | `/auth/me` | ✔ | Current user |
| GET | `/tickets?status=` | ✔ | List (user: own; admin: all); optional status filter |
| POST | `/tickets` | ✔ | Create a ticket (201) |
| GET | `/tickets/{id}` | ✔ | Ticket detail + responses (owner/admin) |
| PATCH | `/tickets/{id}/status` | ✔ admin | Update status (200) |
| POST | `/tickets/{id}/responses` | ✔ admin | Add a response (201) |

HTTP status codes: `200/201/204` success, `401` unauthenticated, `403` forbidden (policy),
`404` not found, `422` validation error.

### Stats service

| Method | Path | Description |
|--------|------|-------------|
| GET | `/health` | Liveness probe |
| GET | `/stats` | `{ total, byStatus, avgResolutionHours, createdLast7Days }` |

## Security notes

- Sanctum tokens hashed (SHA-256) at rest; stored client-side in an **httpOnly** cookie.
- Authorization enforced by `TicketPolicy` (ownership + role), tested end-to-end.
- All input validated via `FormRequest` classes; CORS restricted to `FRONTEND_URL`.
- Auth routes rate-limited (`throttle:auth`). No credentials committed to the repo.
- `APP_DEBUG=true` in local `.env` for convenience — set to `false` in production so stack traces are not exposed.

---

## Known limitations

- Tokens have no refresh/rotation (revocable via logout) — sufficient for the assessment scope.
- No pagination on the ticket list (dataset is small).
- The stats service is read-only and unauthenticated (per brief: the second backend need not
  connect to the main app).
- No real-time updates (websockets) — the UI refetches via TanStack Query invalidation.

## What I'd improve with more time

- Pagination + server-side search/sort on the ticket list.
- End-to-end tests (Playwright) for the frontend and a component test layer.
- Allow customers to reply to their own tickets (currently admin-only, matching the brief).
- Dockerize all three apps for one-command setup; add CI running all test suites.
- Refresh-token rotation and token expiry UX.
